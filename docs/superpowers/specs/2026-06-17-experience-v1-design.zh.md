# 养生预约小程序体验优化 V1 设计稿

## 1. 项目背景

当前项目已经完成第一阶段 MVP：后端接口、后台管理、小程序用户端、预约下单、排班、订单状态流转都已经具备。但第一阶段更偏“功能可用”，页面和流程还停留在功能堆叠层面，用户体验不像成熟的到店按摩、足疗、养生预约产品。

体验优化 V1 的目标不是继续堆功能，而是把现有能力重新组织成更自然的消费决策和门店运营流程。整体方向参考成熟本地生活应用的常见做法：服务套餐化、价格和时长清晰、技师信息可信、可约时间明确、预约入口更短。

本阶段产品定位：

> 单店养生馆/理疗馆的到店预约小程序，主打项目套餐、技师服务、在线预约和后台履约管理。

## 2. 优化目标

### 2.1 用户端目标

用户第一次进入小程序时，不需要解释也能快速完成以下动作：

- 知道这家店提供什么服务。
- 看懂每个项目适合什么人、多少钱、多久。
- 判断该选哪个技师。
- 找到可预约时间。
- 顺利提交预约。
- 下单后知道订单进行到哪一步。

### 2.2 后台端目标

门店工作人员每天打开后台后，可以快速完成：

- 查看今日预约情况。
- 找到待到店、服务中、待处理订单。
- 快速推进订单状态。
- 创建和查看技师排班。
- 减少通过 ID 判断项目和技师的情况。

## 3. 设计原则

1. **以预约转化为核心**
   首页、项目详情、技师详情都要有清晰的预约入口，减少用户在页面之间绕路。

2. **项目先行，技师增强信任**
   对大多数用户来说，先选“肩颈、足疗、艾灸”等项目更自然；技师信息用于增强信任和个性化选择。

3. **每张卡片都要帮助决策**
   列表页不能只展示名称和价格，还要展示时长、适合人群、标签、可预约状态。

4. **后台保持高效，不做花哨展示**
   后台是运营工具，优先保证信息密度、状态清晰、操作路径短。

5. **V1 控制范围**
   暂不做支付、优惠券、会员卡、评价系统、多门店、地图导航等大功能，先把核心预约体验做好。

## 4. 用户端整体体验设计

### 4.1 用户主路径

推荐主路径：

```text
首页
  -> 选择推荐项目 / 分类项目
  -> 查看项目详情
  -> 立即预约
  -> 选择技师
  -> 选择日期和时段
  -> 填写联系人
  -> 提交预约
  -> 查看订单详情
```

辅助路径：

```text
首页
  -> 查看人气技师
  -> 选择技师
  -> 选择服务项目
  -> 选择时间
  -> 提交预约
```

## 5. 小程序页面设计

### 5.1 首页

首页从“信息展示页”调整为“预约转化页”。

#### 顶部门店区

展示内容：

- 门店名称。
- 营业状态。
- 营业时间。
- 门店地址。
- 联系电话。
- 主按钮：`立即预约`。

推荐增加信任标签：

- `正规门店`
- `明码标价`
- `到店服务`
- `可取消预约`

#### 快捷分类入口

展示 4 个常用入口：

- `肩颈放松`
- `经典足疗`
- `艾灸调理`
- `全部项目`

点击分类后进入项目列表，并自动筛选对应类别。

#### 推荐项目

推荐项目卡片展示：

- 项目名称。
- 项目分类。
- 服务时长。
- 售价。
- 适合人群。
- 热门/推荐标签。
- `预约`按钮。

卡片点击进入项目详情；预约按钮直接进入预约页并带上项目 ID。

#### 人气技师

技师卡片展示：

- 头像或姓名首字母。
- 技师姓名。
- 技师等级。
- 从业年限。
- 擅长项目。
- 服务标签。
- 可预约状态。

点击卡片进入技师详情；`预约TA` 直接进入预约页并带上技师 ID。

### 5.2 项目列表页

项目列表从普通列表升级为“套餐选择页”。

#### 筛选区

顶部提供分类切换：

- 全部
- 肩颈
- 足疗
- 艾灸

分类来自现有项目分类数据，默认选中全部。

#### 项目卡片

每张卡片展示：

- 项目名称。
- 分类名称。
- 时长。
- 售价。
- 原价，如有。
- 适合人群。
- 服务亮点标签。
- 热门/推荐标识。
- `查看详情`。
- `预约`。

用户可以不进详情，直接从列表预约，减少路径。

### 5.3 项目详情页

项目详情页从“字段展示页”调整为“购买决策页”。

#### 顶部信息

展示：

- 项目名称。
- 项目分类。
- 服务时长。
- 售价。
- 原价。
- 热门/推荐标签。

#### 内容模块

模块包括：

- `适合人群`
- `服务亮点`
- `服务流程`
- `预约须知`
- `门店保障`

其中：

- `适合人群` 使用现有 suitablePeople 字段。
- `预约须知` 使用现有 notice 字段。
- `服务亮点` 可以从分类、标签、项目文案中整理。
- `门店保障` V1 使用固定文案，例如明码标价、到店服务、预约可取消。

#### 底部操作

固定底部栏展示：

- 当前价格。
- `立即预约`按钮。

点击后进入预约页，并自动选中当前项目。

### 5.4 技师列表页

技师列表从“人员清单”调整为“可信服务人员选择页”。

#### 技师卡片

展示：

- 头像或首字母。
- 姓名。
- 等级。
- 从业年限。
- 擅长项目。
- 服务标签。
- 简短介绍。
- 可预约状态。
- `预约TA`按钮。

如果技师不可约，则弱化预约按钮，并展示原因或状态。

### 5.5 技师详情页

技师详情用于增强用户信任。

展示内容：

- 技师姓名。
- 等级和年限。
- 擅长方向。
- 服务标签。
- 个人介绍。
- 可预约状态。
- `预约TA`按钮。

点击预约后进入预约页，并自动选中当前技师。

### 5.6 预约确认页

预约页从普通表单调整为“分步确认页”。

#### 页面结构

1. 顶部预约摘要。
2. 服务项目选择。
3. 技师选择。
4. 日期选择。
5. 时段选择。
6. 联系人信息。
7. 底部提交按钮。

#### 顶部预约摘要

实时展示用户已经选择的内容：

- 服务项目。
- 技师。
- 日期。
- 时间。
- 价格。

没有选择的内容显示为待选择状态。

#### 服务项目选择

支持从首页、项目列表、项目详情带入项目 ID。

如果没有带入，则默认选择第一个可预约项目，或者提示用户选择。

#### 技师选择

支持从技师列表、技师详情带入技师 ID。

如果没有带入，则默认选择第一个可预约技师。

#### 日期与时段

日期变化后重新加载可约时段。

时段状态：

- 可选。
- 已选。
- 无可约时段。
- 加载中。

空状态文案要告诉用户下一步，例如：

- `请先选择服务项目`
- `当前日期暂无可约时段，可切换其他日期`
- `正在加载可约时段`

#### 联系人信息

字段：

- 联系人姓名。
- 手机号。
- 备注。

提交按钮在必要字段未填完整前置灰。

### 5.7 订单列表页

订单列表从普通列表调整为“预约记录和服务状态页”。

#### 排序规则

优先展示进行中的订单：

1. 已预约。
2. 已到店。
3. 服务中。
4. 已完成。
5. 已取消。

#### 订单卡片

展示：

- 项目名称。
- 技师名称。
- 预约日期。
- 预约时段。
- 订单状态。
- 金额。
- 主要操作。

主要操作按状态变化：

- 已预约：查看详情、取消预约。
- 已完成：再次预约。
- 已取消：再次预约。

### 5.8 订单详情页

订单详情页从信息展示调整为“状态说明页”。

#### 状态进度

展示预约进度：

```text
已预约 -> 已到店 -> 服务中 -> 已完成
```

取消订单展示取消状态，不再显示后续流程。

#### 详情信息

展示：

- 服务项目。
- 服务技师。
- 预约时间。
- 联系人。
- 手机号。
- 订单金额。
- 用户备注。

#### 操作区

根据状态展示：

- 已预约：取消预约。
- 已完成：再次预约。
- 已取消：再次预约。

同时提供门店电话入口，方便用户联系门店。

## 6. 后台管理体验设计

后台本阶段不做大重构，只做运营效率优化。

### 6.1 数据看板

看板从简单统计升级为今日运营概览。

核心指标：

- 今日预约数。
- 待到店。
- 服务中。
- 已完成。
- 已取消。
- 可预约技师数。
- 上架项目数。

增加一个“今日待处理预约”区域，展示需要工作人员操作的订单。

### 6.2 预约订单页

预约订单页从数据库表格调整为运营表格。

#### 列表展示

减少直接展示 ID：

- 项目 ID 改成项目名称。
- 技师 ID 改成技师名称。

保留订单 ID，但弱化展示。

#### 筛选能力

增加状态筛选：

- 全部。
- 已预约。
- 已到店。
- 服务中。
- 已完成。
- 已取消。

#### 操作按钮

当前每行展示多个按钮，很多都是禁用状态，操作成本高。

优化为只展示当前状态的下一步操作：

- 已预约：到店、取消。
- 已到店：开始服务、取消。
- 服务中：完成服务。
- 已完成：无主操作。
- 已取消：无主操作。

取消操作保留确认弹窗。

### 6.3 排班管理页

排班页优化重点是表单清晰。

优化内容：

- 日期、技师、排班类型、开始时间、结束时间靠近展示。
- 排班类型展示中文：上班、休息、请假、占用。
- 空状态提示更明确。
- 后端返回冲突或错误时，前端展示用户能理解的错误文案。

## 8. 异常与空状态设计

### 8.1 用户端

常见异常：

- 首页加载失败。
- 项目列表为空。
- 技师列表为空。
- 某日期无可约时段。
- 预约提交失败。
- 订单加载失败。

处理方式：

- 提供明确原因。
- 保留用户已填写内容。
- 提供重试或下一步建议。

示例：

- `当前日期暂无可约时段，可切换其他日期`
- `门店暂未配置可预约技师，请稍后再试`
- `预约提交失败，请检查手机号或稍后重试`

### 8.2 后台端

常见异常：

- 登录过期。
- 列表加载失败。
- 状态流转失败。
- 排班创建失败。

处理方式：

- 登录过期继续跳转登录页。
- 操作失败用消息提示。
- 状态更新失败时不提前改 UI 状态。
- 排班失败展示后端返回的业务原因。

## 9. 验收标准

### 9.1 小程序验收

必须完成以下链路：

1. 首页能看到门店、推荐项目、人气技师、快捷分类。
2. 从首页点击推荐项目能进入项目详情。
3. 从项目详情点击立即预约能进入预约页，并自动选中项目。
4. 从技师详情点击预约TA能进入预约页，并自动选中技师。
5. 预约页能完成项目、技师、日期、时段、联系人选择。
6. 提交预约后进入订单详情。
7. 订单详情能展示状态进度和预约摘要。
8. 已预约订单可以取消。
9. 已完成或已取消订单可以再次预约。

### 9.2 后台验收

必须完成以下链路：

1. 登录后台成功。
2. 数据看板展示今日预约状态。
3. 预约订单页能按日期和状态筛选。
4. 预约订单页展示项目名称和技师名称，或者至少在前端映射为名称。
5. 不同订单状态只展示合理的下一步操作。
6. 可以完成到店、开始服务、完成服务、取消预约。
7. 排班页可以创建上班排班。

## 10. 实施顺序

建议按以下顺序实施：

1. 小程序公共展示工具和状态文案整理。
2. 首页体验重构。
3. 项目列表和项目详情优化。
4. 技师列表和技师详情优化。
5. 预约确认页重构。
6. 订单列表和订单详情优化。
7. 后台看板和预约订单页优化。
8. 排班页轻量优化。
9. 更新 smoke test 文档。
10. 完整验证。

## 11. V1 不做的内容

为了保证体验优化能落地，以下内容本阶段不做：

- 微信支付。
- 优惠券。
- 会员卡。
- 疗程卡。
- 用户评价系统。
- 多门店。
- 地图导航。
- 技师绩效。
- 客户 CRM。
- 营销活动配置。

这些功能可以作为 V2 或后续商业化阶段再设计。

## 12. 总结

体验优化 V1 的核心不是把系统做大，而是把现有 MVP 做顺。

小程序端要从“功能入口集合”变成“预约决策工具”；后台端要从“数据表管理”变成“门店今日运营工具”。

这一版完成后，项目会更接近真实养生馆、足疗馆、理疗馆能拿去试运营的状态。
